Problem Definition

문제 정의

"모두 정상"이라고 예측해도 정확도 98.48%가 나오는 극단적 불균형 환경에서, 정확도 대신 부도 기업을 놓치지 않는 것(FN 최소화)을 핵심 목표로 설정했습니다.

이 프로젝트는 "부도 기업을 얼마나 많이 맞히는가"보다 부도 기업을 놓치지 않는가에 초점을 둔 조기경보 문제로 정의했습니다. 전체 50,000개 기업 중 부도는 758개뿐이어서, 정확도는 쉽게 높아질 수 있지만 실제 금융 리스크를 줄이는 데는 거의 도움이 되지 않았습니다.

단순 실험 참여를 넘어, 분석 방향 설계와 의사결정에 직접 기여했습니다.

역할

피처 엔지니어링 방향 설계 159개 원본 변수 기반 수동 선택과 중요도 기반 선택을 직접 시도하고, 한계를 확인한 뒤 도메인 기반 접근 전환을 주도했습니다.
실험 참여 및 결과 해석 SMOTE 등 불균형 처리, 모델별 PR-AUC 비교, 임계값 실험 결과를 해석하고 최종 의사결정에 참여했습니다.
발표 및 스토리텔링 FN 중심 리스크 판단, Streamlit 서비스 구조, SHAP 기반 설명 가능성을 발표 메시지로 구조화했습니다.

핵심 기여 포인트

  • 원본 변수 직접 선별 실험 후 "숫자 나열만으로는 부도 메커니즘을 설명하기 어렵다"는 한계 인식
  • 멘토의 조언을 반영해 도메인 기반 Feature Engineering으로 전환
  • 부도 3대 경로(유동성 위기 · 지급불능 · 재무조작)를 중심으로 52개 피처 설계 기여
  • SMOTE, Borderline-SMOTE, Class Weight 등 비교 후 Recall 중심 운영에 적합한 실험 방향 도출
  • Data Leakage 변수(이해관계자_불신지수)를 직접 식별·제거
  • Validation과 CV 결과를 함께 반영해 임계값을 평균 산정, 과적합 리스크 완화
Decision Making

차별화된 의사결정

이 프로젝트의 강점은 모델 점수보다도, 왜 그런 선택을 했는지 설명할 수 있다는 점입니다.

왜 Recall을 최우선으로?

부도 기업을 정상으로 놓치면(FN) 대출 손실이 직접 발생하지만, 정상 기업을 부도로 예측하면(FP) 심사를 한 번 더 하면 됩니다. 비용 구조가 비대칭이므로 Recall 중심 설계가 유일한 선택이었습니다.

왜 CatBoost 단일 모델?

Voting Ensemble(0.1463)이 근소하게 높았지만, Wilcoxon Test p=0.0625로 통계적 차이가 없었습니다. 복잡도 증가 대비 이득이 없으므로 해석 가능하고 유지보수가 쉬운 단일 모델을 선택했습니다.

왜 임계값 0.0856?

단일 Validation 결과에 과적합되지 않도록 Validation(0.0940)과 CV(0.0772) 평균으로 산정했습니다. 운영 환경에서 일관된 성능을 보장하기 위한 선택이었습니다.

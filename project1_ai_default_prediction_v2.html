역할

Challenges

고민했던 과정

단순 실행이 아니라, 왜 이 선택을 했는지 설명할 수 있는 부분들입니다.

불균형 데이터를 어떻게 다룰까

SMOTE, Borderline-SMOTE, SMOTE-Tomek, Class Weight를 비교 실험하며 오버샘플링 방법이 SMOTE 외에도 다양한 변형이 존재한다는 걸 배웠습니다.

1:65 불균형 데이터에서는 "모두 정상"으로만 예측해도 Accuracy 98%가 나와 평가 지표로 의미가 없었습니다. ROC-AUC도 TNR(다수 클래스를 얼마나 잘 맞히는지)이 반영돼 점수가 부풀려졌습니다. 반면 PR-AUC는 Precision·Recall만으로 계산되어 다수 클래스 예측 성능과 무관하게 부도 탐지력만 보여준다는 것을 실험을 통해 배웠고, 이를 핵심 평가 지표로 삼았습니다.

또한 Recall을 높이기 위해 임계값을 낮추면 FP가 폭증해 Precision이 5%대까지 떨어지는 문제가 있었습니다. "부도 기업을 놓치지 않겠다"는 목표와 "오탐이 너무 많다"는 현실 사이에서 어느 쪽을 우선할지 직접 판단해야 했습니다. 최종 임계값 0.0856은 FN 비용이 FP보다 훨씬 크다는 비용 구조에 맞게 설계된 운영 기준입니다.

피처를 어떻게 선택할까

도메인 지식 없이 수동으로 변수를 고르다 한계를 느꼈습니다. Feature Importance 순위가 높다는 것이 인과관계를 의미하지 않는다는 점도 깨달았습니다. 결국 부도 메커니즘을 먼저 이해하고 도메인 구조로 피처를 설계하는 방향으로 전환했습니다.

모델을 어떻게 서비스화할까

좋은 모델도 쓸 수 없으면 의미가 없다고 생각했습니다. Streamlit을 선택해 DART API 연동 기업 검색과 SHAP 기반 리스크 리포트를 제공하는 웹 서비스를 직접 배포했습니다.

Lessons Learned

배운 점

만족스럽지 않은 결과였지만, 그 과정에서 얻은 것들입니다.

Feature Importance는 인과관계가 아님

중요도 순위가 높다는 것은 모델이 그 변수를 예측에 활용한다는 뜻이지, 부도를 유발한다는 의미가 아닙니다. SHAP으로 방향성까지 함께 봐야 한다는 것을 배웠습니다.

Quantity보다 Quality — 피처 압축의 중요성

VIF로 다중공선성을 제거해도 피처가 많으면 모델이 강한 신호 하나에만 의존하게 됩니다. PCA나 군집화로 핵심 구조를 먼저 파악하는 게 더 효과적이었을 것 같습니다.

SMOTE는 불균형의 근본 해결이 아님

SMOTE로 합성한 소수 클래스가 진짜 부도 패턴을 담지 못한다는 한계를 실험을 통해 느꼈습니다. 극단적 불균형에서는 샘플링 기법보다 비용 구조를 반영한 평가 지표 설계가 더 중요했을 것 같습니다.

예측은 기술이 아니라 도메인 이해에서 시작됨

숫자를 잘 다루는 것보다 왜 부도가 나는가를 먼저 이해해야 의미 있는 피처를 만들 수 있었습니다. 이후에는 심층 EDA로 데이터 본질을 탐구하는 방향으로 성장하고 싶습니다.

Problem Definition

문제 정의

"모두 정상"이라고 예측해도 정확도 98.48%가 나오는 극단적 불균형 환경에서, 정확도 대신 부도 기업을 놓치지 않는 것(FN 최소화)을 핵심 목표로 설정했습니다.

이 프로젝트는 "부도 기업을 얼마나 많이 맞히는가"보다 부도 기업을 놓치지 않는가에 초점을 둔 조기경보 문제로 정의했습니다. 전체 50,000개 기업 중 부도는 758개뿐이어서, 정확도는 쉽게 높아질 수 있지만 실제 금융 리스크를 줄이는 데는 거의 도움이 되지 않았습니다.

Decision Making

차별화된 의사결정

이 프로젝트의 강점은 모델 점수보다도, 왜 그런 선택을 했는지 설명할 수 있다는 점입니다.

왜 Recall을 최우선으로?

부도 기업을 정상으로 놓치면(FN) 대출 손실이 직접 발생하지만, 정상 기업을 부도로 예측하면(FP) 심사를 한 번 더 하면 됩니다. 비용 구조가 비대칭이므로 Recall 중심 설계가 유일한 선택이었습니다.

왜 CatBoost 단일 모델?

Voting Ensemble(0.1463)이 근소하게 높았지만, Wilcoxon Test p=0.0625로 통계적 차이가 없었습니다. 복잡도 증가 대비 이득이 없으므로 해석 가능하고 유지보수가 쉬운 단일 모델을 선택했습니다.

왜 임계값 0.0856?

단일 Validation 결과에 과적합되지 않도록 Validation(0.0940)과 CV(0.0772) 평균으로 산정했습니다. 운영 환경에서 일관된 성능을 보장하기 위한 선택이었습니다.
